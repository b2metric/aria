Bring Your Own Key (BYOK)

Data sovereignty and custom LLM routing for enterprise customers.

What is BYOK?

Bring Your Own Key (BYOK) is an enterprise security feature that allows customers to provide their own Large Language Model (LLM) API keys (such as OpenAI, Azure OpenAI, or Google Gemini) instead of using the shared ARIA platform key.

This ensures that sensitive queries and data schemas are sent directly to the customer's dedicated model endpoints, fulfilling strict data sovereignty and compliance requirements (e.g., GDPR, KVKK) often demanded by banking and healthcare organizations.

How it Works

Even when a customer brings their own key, ARIA maintains full control over governance, token accounting, and security through its proxy architecture.

  1. Proxy Routing: Queries are still routed through the ARIA LiteLLM proxy. This allows ARIA to count tokens, log audits, and enforce quotas.
  2. Token Quotas: Custom keys do not bypass usage limits. Token consumption is intercepted from the provider's response payload, and the customer's daily TokenQuota is strictly enforced.
  3. Encrypted Storage: All customer API keys are encrypted at rest using AES/Fernet encryption via the CustomerLLMConfig table and decrypted only dynamically in memory before execution.

Configuring BYOK

Workspace Administrators can configure their dedicated LLM provider through the ARIA Admin interface:

Setup Path: Navigation Menu ➔ LLM Config (BYOK)
  • Select the LLM provider (OpenAI, Azure, Anthropic, Gemini).
  • Provide the endpoint API Base (if using Azure or a self-hosted enterprise endpoint).
  • Supply the secret API Key (this is masked and encrypted upon saving).
  • Specify the Model Name or Azure Deployment Version (e.g., gpt-4o).

Fallback Mechanisms

If a customer disables their BYOK configuration, or if their API key expires/fails authentication repeatedly, ARIA's llm_resolver.py will automatically fall back to the platform's default shared key (if allowed) to ensure continuity of service.
