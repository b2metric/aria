CMEK & Envelope Encryption

To support strict enterprise compliance and data residency requirements, ARIA implements Envelope Encryption for all sensitive credentials (like Database passwords and external LLM API keys) stored at rest.

How Envelope Encryption Works

Instead of using a single global key to encrypt all tenant passwords, ARIA uses a two-tier key hierarchy:

  1. Data Encryption Key (DEK): Every tenant/workspace is assigned a unique, randomly generated 32-byte DEK. All credentials belonging to that tenant are encrypted using this DEK.
  2. Key Encryption Key (KEK): The tenant's DEK is itself encrypted (wrapped) by a master KEK.

Customer Managed Encryption Keys (CMEK)

By default, the KEK is provided by the ARIA application layer (App KEK). However, enterprise customers can choose to integrate their own cloud provider's Key Management Service (KMS) to wrap their DEK.

Supported KMS Providers:
  • AWS Key Management Service (KMS)
  • Google Cloud KMS
  • Azure Key Vault

Revocation & Access

Because the DEK is wrapped by a customer-controlled KMS key, the customer can instantly revoke ARIA's access to their databases and LLMs by disabling or deleting the key in their own AWS/GCP/Azure console. Once the KEK is revoked, ARIA can no longer unwrap the DEK, effectively crypto-shredding the tenant's credentials.

Configuration

Customers can manage their CMEK configuration in the ARIA UI by navigating to Workspace Settings > Encryption.
